# PRD: 邀请返利与签到奖励金额化

> 状态：V4 实施验收版  
> 更新日期：2026-06-24  
> 审查参考：`docs/prd/invitation-rebate-checkin-amount-prd-review.md`

## 1. 结论

把邀请奖励、充值返利、签到奖励、返利转余额的产品口径统一为 USD 金额。后台配置、用户输入、日志和页面展示都按金额表达；内部继续写入现有 `quota` 整数字段，以兼容现有计费、扣费、余额判断和历史数据。

本 PRD 已吸收审查报告中的关键修正：

| 审查问题 | 最终决策 |
| --- | --- |
| `TopUp.Money` 在不同渠道语义不一致 | 新增并使用 `topups.original_pay_amount_usd` 作为充值返利唯一基数 |
| 充值返利会侵入多个支付完成路径 | 新增统一 `AfterTopUpSuccessHook(topUp)`，支付路径只接入钩子 |
| Epay 回调事务边界特殊 | 充值到账优先；返利独立事务；Epay 至少在 `LockOrder` 保护区内调用钩子并依赖 DB 幂等 |
| 首充判断缺少可靠依据 | 新增 `users.has_first_topup_rebate`，表示首充返利机会是否已消费 |
| `aff_count` 语义变更风险 | `aff_count` 继续表示邀请注册人数；新增 `aff_rebate_count` 表示邀请人获得充值返利次数 |
| 重复 webhook / 补单风险 | 新增 `topup_rebate_logs`，用唯一索引保证同一订单同一接收人不重复发放 |

## 2. 背景与目标

当前系统中邀请奖励和签到奖励主要以内部 `quota` 配置和发放。管理员看到的是 token/quota 类数值，但充值、余额和账务页面更接近金额语义，导致配置成本不直观、页面展示口径不统一。

本次目标：

- 邀请返利支持三种触发模式：注册后发放、首次充值后发放、每次充值都发放。
- 注册奖励使用固定 USD 金额。
- 充值返利使用原始实际支付金额乘以返利百分比。
- 邀请人和受邀人分别配置奖励金额或返利百分比。
- 签到奖励按 USD 金额配置最小值和最大值。
- 用户侧返利、签到、转余额均按 USD 金额展示和输入。
- 旧 quota 字段和旧接口字段继续保留，避免旧前端或第三方调用断裂。
- 不改变核心余额扣费、模型计费和余额不足判断逻辑。

## 3. 非目标

- 不做全站数据库余额单位迁移。
- 不删除 `quota`、`aff_quota`、`aff_history_quota`、`quota_awarded`、`aff_count` 等旧字段。
- 不把订阅购买计入充值返利。
- 不给历史成功充值自动补发新规则下的返利。
- 不把返利失败设计成充值到账失败。
- 不复用 `aff_count` 表示充值返利次数。

## 4. 术语

| 术语 | 定义 |
| --- | --- |
| 主余额 | 用户可直接消费的余额，内部字段为 `quota` |
| 待返利余额 | 邀请人获得但尚未转入主余额的返利，内部字段为 `aff_quota` |
| 累计返利 | 邀请人历史获得的返利累计，内部字段为 `aff_history_quota` |
| 邀请人数 | 成功绑定邀请关系的注册人数，字段为 `aff_count` |
| 充值返利次数 | 邀请人实际获得充值返利的次数，字段为 `aff_rebate_count` |
| 原始支付金额 | 用户在支付渠道实际支付的 USD 金额，不包含分组倍率、赠送额度或内部换算后的余额 |
| 钱包充值 | 购买主余额的充值订单，不包含订阅购买 |

## 5. 核心规则

### 5.1 邀请返利触发模式

新增配置项：`quota_setting.affiliate_reward_trigger`。

| 值 | 后台展示 | 行为 |
| --- | --- | --- |
| `registration` | 注册后发放 | 受邀用户注册成功并绑定邀请关系后，立即按固定金额发放奖励 |
| `first_topup` | 首次充值后发放 | 受邀用户第一笔成功钱包充值后，按充值金额百分比发放一次返利 |
| `every_topup` | 每次充值都发放 | 受邀用户每笔成功钱包充值后，按充值金额百分比发放返利 |

默认值为 `registration`，保持现有“注册即奖励”的默认体验。

### 5.2 注册后发放

配置项：

| 配置项 | 含义 |
| --- | --- |
| `quota_setting.inviter_registration_reward_amount` | 邀请人固定奖励金额，单位 USD |
| `quota_setting.invitee_registration_reward_amount` | 受邀人固定奖励金额，单位 USD |

规则：

- 普通注册和 OAuth 注册均适用。
- 仅在存在有效邀请人时绑定邀请关系。
- 邀请关系绑定成功后，`aff_count` 递增。
- 只有 `affiliate_reward_trigger=registration` 时发放固定奖励。
- 非注册触发模式下，注册只绑定邀请关系，不发放邀请奖励。
- 邀请人奖励进入 `aff_quota` 和 `aff_history_quota`。
- 受邀人奖励进入主余额 `quota`。
- 奖励金额为 0 时不入账，但不影响注册和邀请关系绑定。

### 5.3 充值后发放

配置项：

| 配置项 | 含义 |
| --- | --- |
| `quota_setting.inviter_topup_rebate_percent` | 邀请人充值返利百分比 |
| `quota_setting.invitee_topup_rebate_percent` | 受邀人充值返利百分比 |

计算规则：

```text
rebate_amount_usd = original_pay_amount_usd * rebate_percent / 100
rebate_quota = round(rebate_amount_usd * QuotaPerUnit)
```

发放规则：

- `first_topup` 只处理受邀用户第一笔成功钱包充值。
- `every_topup` 处理受邀用户每笔成功钱包充值。
- 邀请人返利进入 `aff_quota` 和 `aff_history_quota`。
- 受邀人返利进入主余额 `quota`。
- 邀请人返利成功入账后，`aff_rebate_count` 递增。
- 邀请人百分比为 0 时，不给邀请人发放，也不增加 `aff_rebate_count`。
- 受邀人百分比为 0 时，不给受邀人发放。
- `first_topup` 模式下，即使百分比为 0，该用户的首充返利机会也视为已消费，后续不因配置变化追溯补发。
- 百分比允许大于等于 0；是否限制上限属于运营策略，本 PRD 不强制限制为 100。

触发范围：

- Epay 钱包充值
- Stripe 钱包充值
- Creem 钱包充值
- Waffo 钱包充值
- Waffo Pancake 钱包充值
- 管理员补单钱包充值

排除范围：

- 订阅购买
- 非钱包充值订单
- 未成功支付或已取消订单

## 6. 金额与内部额度换算

统一使用后端 helper 完成换算：

```text
internal_quota = round(amount_usd * QuotaPerUnit)
amount_usd = internal_quota / QuotaPerUnit
```

要求：

- 金额计算使用 decimal 或等价的十进制计算方式，避免二进制浮点误差。
- 金额到 quota 的四舍五入必须集中封装，禁止在业务路径散落重复换算。
- 注册奖励、充值返利、签到奖励、返利转余额必须共用同一换算口径。
- 配置和接口金额字段允许小数。
- 前端默认展示 2 位小数。
- 审计日志可保留最多 6 位小数，用于支付渠道金额精度。

## 7. 数据模型与配置

### 7.1 新增配置项

| 配置项 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `quota_setting.affiliate_reward_trigger` | string | `registration` | 邀请奖励触发模式 |
| `quota_setting.inviter_registration_reward_amount` | number | 旧 `QuotaForInviter` 反算金额或 0 | 邀请人注册奖励金额 |
| `quota_setting.invitee_registration_reward_amount` | number | 旧 `QuotaForInvitee` 反算金额或 0 | 受邀人注册奖励金额 |
| `quota_setting.inviter_topup_rebate_percent` | number | 0 | 邀请人充值返利百分比 |
| `quota_setting.invitee_topup_rebate_percent` | number | 0 | 受邀人充值返利百分比 |
| `checkin_setting.min_amount` | number | 旧 `min_quota` 反算金额或 0 | 签到最小奖励金额 |
| `checkin_setting.max_amount` | number | 旧 `max_quota` 反算金额或 0 | 签到最大奖励金额 |

校验规则：

- 触发模式只能是 `registration`、`first_topup`、`every_topup`。
- 金额必须大于等于 0。
- 百分比必须大于等于 0。
- 签到最大奖励金额不得小于最小奖励金额。
- 非零邀请奖励或充值返利百分比仍需满足当前系统的支付合规确认要求。

### 7.2 新增数据库字段

| 表 | 字段 | 说明 |
| --- | --- | --- |
| `topups` | `original_pay_amount_usd` | 充值返利使用的原始实际支付金额 |
| `users` | `has_first_topup_rebate` | 首充返利机会是否已消费 |
| `users` | `aff_rebate_count` | 邀请人获得充值返利次数 |
| `topup_rebate_logs` | 见 8.2 | 充值返利幂等与审计日志 |

数据库要求：

- SQLite、MySQL、PostgreSQL 均需兼容。
- 新字段允许旧数据平滑升级。
- 唯一索引必须可在三种数据库中创建。
- 不使用数据库方言独有能力作为唯一实现路径。

### 7.3 保留旧字段

继续保留：

- `quota_awarded`
- `aff_quota`
- `aff_history_quota`
- `aff_count`
- 旧配置 `QuotaForInviter`
- 旧配置 `QuotaForInvitee`
- 旧签到配置 `checkin_setting.min_quota`
- 旧签到配置 `checkin_setting.max_quota`

### 7.4 新增响应金额字段

推荐补充：

- `quota_amount`
- `quota_awarded_amount`
- `total_quota_amount`
- `aff_quota_amount`
- `aff_history_quota_amount`
- `aff_rebate_count`

兼容规则：

- 旧字段不改名、不删除。
- 新前端默认使用金额字段。
- 旧前端仍可继续读取 quota 字段。
- 第三方调用方不会因新增字段而破坏兼容。

## 8. 充值返利架构

### 8.1 统一成功钩子

新增统一入口：

```text
AfterTopUpSuccessHook(topUp)
```

职责：

- 判断订单是否为成功的钱包充值。
- 读取 `affiliate_reward_trigger`。
- 在 `first_topup` 或 `every_topup` 模式下处理充值返利。
- 使用 `topups.original_pay_amount_usd` 计算返利。
- 写入幂等日志和余额变更。
- 记录失败日志，支持人工核对与补偿。

接入原则：

- 支付完成路径只负责确认订单成功和用户充值到账。
- 支付完成路径不内联邀请返利业务。
- 钱包充值成功后调用统一钩子。
- 订阅购买路径不调用该钩子。

事务边界：

- 用户充值到账优先级高于邀请返利。
- 充值到账事务成功后，再调用返利钩子。
- 返利钩子使用独立事务。
- 返利失败不得回滚用户充值到账。
- 返利失败必须记录订单号、用户 ID、邀请人 ID、触发模式和错误原因。

Epay 特别要求：

- Epay 当前回调路径存在进程内锁和非统一事务边界，是最高风险接入点。
- 优先目标是把 Epay 成功处理收敛到与其他渠道一致的事务和状态检查模式。
- 如果短期不重构，至少必须在 `LockOrder` 保护区内调用钩子，并依赖数据库幂等日志兜底。

### 8.2 返利日志与幂等

新增表：`topup_rebate_logs`。

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `trade_no` | 充值订单号 |
| `topup_id` | 充值订单 ID |
| `invitee_id` | 受邀用户 ID |
| `recipient_user_id` | 返利接收用户 ID |
| `recipient_role` | `inviter` 或 `invitee` |
| `trigger_mode` | `first_topup` 或 `every_topup` |
| `original_pay_amount_usd` | 原始实际支付金额 |
| `rebate_percent` | 本次使用的返利百分比 |
| `rebate_amount_usd` | 本次返利金额 |
| `rebate_quota` | 本次返利内部额度 |
| `created_at` | 创建时间 |

唯一约束：

```text
trade_no + recipient_user_id + recipient_role
```

幂等规则：

- 邀请人和受邀人分别写独立日志。
- 同一订单可以给邀请人和受邀人各发一条返利。
- 同一订单不得给同一接收人重复发放。
- 日志写入和余额更新必须在同一事务内完成。
- 唯一约束冲突表示该订单对该接收人的返利已完成，直接跳过。
- 余额更新失败时，返利事务回滚，日志也回滚，后续可重试。
- `aff_rebate_count` 只在邀请人返利日志创建成功且余额入账成功后递增。

### 8.3 首充判断

新增字段：`users.has_first_topup_rebate`。

规则：

- 默认值为 `false`。
- 仅表示“首充返利机会是否已消费”，不是“用户是否充值过”的通用字段。
- `first_topup` 模式下，成功钱包充值进入钩子后，在返利事务内检查并更新该字段。
- 当字段为 `true` 时，后续充值跳过首充返利。
- 并发充值或重复回调必须通过事务检查和返利日志共同保证最多处理一次。

### 8.4 返利基数采集

统一使用 `topups.original_pay_amount_usd`。

| 路径 | 要求 |
| --- | --- |
| Epay | 创建订单或回调确认时写入用户实际支付金额 |
| Stripe | 使用支付事件中的实际支付金额，不使用分组倍率后的余额金额 |
| Creem | 使用支付事件中的实际支付金额 |
| Waffo | 使用订单实际支付金额 |
| Waffo Pancake | 使用订单实际支付金额 |
| 管理员补单 | 使用补单时录入或订单保存的实际支付金额 |

缺失处理：

- 新订单应尽量在订单创建阶段写入该字段。
- 支付确认阶段如果发现字段缺失且渠道可提供金额，应补写。
- 历史订单缺少该字段时，不自动补发充值返利。
- 充值返利逻辑不得直接把 `TopUp.Money` 当作统一基数。

## 9. 签到奖励金额化

配置项：

| 配置项 | 含义 |
| --- | --- |
| `checkin_setting.min_amount` | 单次签到最小奖励金额，单位 USD |
| `checkin_setting.max_amount` | 单次签到最大奖励金额，单位 USD |

规则：

- 签到时在 `[min_amount, max_amount]` 范围内随机金额。
- 随机金额转换为内部 `quota` 后入账。
- 签到记录继续保留 `quota_awarded`。
- 签到响应和统计补充金额字段。
- 同一用户同一天只能签到一次的规则不变。
- 保存配置时应拒绝 `max_amount < min_amount`。

推荐响应字段：

| 字段 | 含义 |
| --- | --- |
| `quota_awarded_amount` | 本次签到奖励金额 |
| `total_quota_amount` | 统计口径下的累计签到金额 |

## 10. 邀请奖励转余额

转余额改为金额语义。

前端行为：

- 用户输入 USD 金额。
- 弹窗显示可转金额。
- 默认不再展示 token/quota 文案。

后端行为：

- 新请求字段使用 `amount`。
- 后端把 `amount` 转为内部 `quota`。
- 校验转换后的额度不能超过 `aff_quota`。
- 成功后减少 `aff_quota`，增加主余额 `quota`。
- 保留旧 `quota` 请求字段兼容旧前端。

推荐请求体：

```json
{
  "amount": 5.25
}
```

兼容规则：

- 如果请求包含 `amount > 0`，优先使用金额语义。
- 如果没有 `amount`，但包含旧字段 `quota > 0`，按旧逻辑处理。
- 响应继续返回 `aff_quota` 等旧字段，同时补充金额字段。

## 11. 老配置兼容与迁移

邀请注册奖励：

- 如果新金额配置存在，优先使用新金额配置。
- 如果新金额配置不存在，但旧 `QuotaForInviter` 或 `QuotaForInvitee` 存在，则按 `quota / QuotaPerUnit` 反算金额。
- 管理员保存设置后，写入新金额配置。
- 保存新金额配置时可以同步写入旧 quota 配置，兼容旧逻辑。
- 默认前端不再展示旧 quota 配置入口。

签到奖励：

- 如果 `min_amount` 和 `max_amount` 存在，优先使用金额配置。
- 如果新金额配置不存在，但旧 `min_quota` 或 `max_quota` 存在，则按 `quota / QuotaPerUnit` 反算金额。
- 管理员保存设置后，写入新金额配置。

历史数据：

- 历史 `aff_quota` 和 `aff_history_quota` 不迁移，只在展示时按 `QuotaPerUnit` 反算金额。
- 历史签到记录不强制回填金额字段，可以在响应时按 `quota_awarded` 反算。
- 历史成功充值不自动补发充值返利。

## 12. 前后台体验

### 12.1 后台设置

邀请奖励设置：

- 新增“邀请奖励触发时机”。
- 选项为“注册后发放”“首次充值后发放”“每次充值都发放”。
- 选择“注册后发放”时，展示邀请人和受邀人固定奖励金额输入。
- 选择“首次充值后发放”或“每次充值都发放”时，展示邀请人和受邀人返利百分比输入。
- 金额输入明确标注 USD。
- 百分比输入明确标注 `%`。
- 默认不再让管理员按 token/quota 配置邀请奖励。

签到设置：

- 将签到奖励范围从 quota 改为 USD 金额。
- 展示最小签到奖励金额和最大签到奖励金额。
- 如果来自旧 quota 配置回退，页面展示换算后的金额。
- 保存后落到新金额配置。

### 12.2 用户侧

钱包邀请奖励卡片展示：

- 邀请人数
- 待返利金额
- 累计返利金额
- 充值返利次数

转余额弹窗：

- 输入 USD 金额。
- 显示可转金额。
- 提交 `amount` 字段。
- 不再出现 token/quota 文案。

签到页面：

- 本次签到奖励显示金额。
- 月累计和总累计显示金额。
- 历史记录可保留 quota 字段兼容，但默认展示金额。

## 13. 日志、补偿与可观测性

必须记录的信息：

- 充值订单号
- 支付渠道
- 受邀用户 ID
- 邀请人 ID
- 触发模式
- 原始支付金额
- 返利百分比
- 返利金额
- 返利 quota
- 幂等命中或失败原因

补偿要求：

- 返利钩子失败不影响充值到账。
- 失败日志必须足够支持人工核对。
- 管理员补单再次触发时，不得绕过幂等日志重复发放。
- 人工补偿前必须能区分“从未发放”和“已发放但本次请求幂等跳过”。

## 14. 发布与回滚

推荐分阶段上线：

1. 上线金额换算 helper、配置读取兼容和新增数据字段。
2. 上线前端配置项，默认触发模式保持 `registration`。
3. 接入统一充值成功钩子，并在默认模式下保持充值返利不触发。
4. 完成支付渠道回归后，允许管理员切换到 `first_topup` 或 `every_topup`。
5. 观察充值回调、返利日志和用户余额变化后再扩大使用。

回滚策略：

- 将 `affiliate_reward_trigger` 改回 `registration` 可停止新增充值返利。
- 新增字段保留，不影响旧余额扣费。
- 已发放奖励不自动回滚，如需处理走人工账务调整。

上线 Gate：

- `topup_rebate_logs` 已创建并验证唯一约束生效。
- 所有钱包充值成功路径已接入统一钩子。
- Epay 重复通知专项测试通过。
- 旧配置到新金额字段的回退逻辑通过。
- 前端 en、zh、fr、ja、ru、vi 翻译无缺失 key。
- 后端测试、前端 typecheck 和前端 build 通过。

## 15. 验收标准

### 15.1 产品验收

- 管理员可以保存三种邀请返利触发模式。
- 注册模式下，普通注册和 OAuth 注册都会按固定金额处理邀请奖励。
- 首充模式下，注册只绑定邀请关系，第一笔成功钱包充值发放一次返利，第二笔不重复发。
- 每次充值模式下，每笔成功钱包充值都按原始实际支付金额百分比返利。
- `aff_count` 始终表示邀请注册人数。
- `aff_rebate_count` 只随邀请人成功获得充值返利递增。
- 同一订单重复回调或重复补单不会重复返利。
- 订阅购买不触发充值返利。
- 签到奖励按金额范围随机并入账。
- 转余额按金额输入，不能超过待返利金额。
- 前端新增文案完成 en、zh、fr、ja、ru、vi 翻译。

### 15.2 技术验收

- Epay、Stripe、Creem、Waffo、Waffo Pancake、管理员补单均通过统一钩子触发充值返利。
- 返利基数来自 `original_pay_amount_usd`。
- 返利日志唯一约束可以防止同一订单同一接收人重复发放。
- 返利事务失败时，日志和余额更新一起回滚。
- 充值到账成功后，即使返利失败也不回滚充值。
- SQLite、MySQL、PostgreSQL 都能完成迁移和启动。
- 旧配置可以换算为金额展示和运行。
- 旧 API 字段仍存在。

## 18. 风险清单

| 风险 | 等级 | 处理方式 |
| --- | --- | --- |
| 返利基数使用错误 | P0 | 强制使用 `original_pay_amount_usd`，禁止把 `TopUp.Money` 当统一基数 |
| 重复回调导致重复返利 | P0 | 返利日志唯一约束和事务保护 |
| Epay 非统一事务路径 | P0 | 优先重构；短期在 `LockOrder` 内接入并依赖 DB 幂等兜底 |
| 首充并发重复发放 | P1 | `has_first_topup_rebate` 与返利日志共同约束 |
| `aff_count` 语义混乱 | P1 | 保持旧语义，新增 `aff_rebate_count` |
| 老配置升级后丢失 | P1 | 新字段缺失时从旧 quota 配置反算金额 |
| 返利失败影响充值到账 | P1 | 返利事务独立，失败只记录日志 |
| 前端金额和后端 quota 不一致 | P1 | 所有换算走统一 helper，并返回金额字段供展示 |

## 19. 最终决策

本需求可以实施，但正式开启 `first_topup` 或 `every_topup` 前必须满足：

- 充值返利基数统一为原始实际支付金额。
- 所有钱包充值成功路径接入统一钩子。
- 返利幂等日志和唯一约束已上线并验证。
- `aff_count` 语义保持不变。
- 旧配置和旧 API 字段保持兼容。
- Epay 路径完成重复回调专项回归。
